# Research spike outcomes

The solution design pack defined 15 research spikes (SPIKE-001 … SPIKE-015).
Most were resolved by building and operating the application rather than as
stand-alone studies; this file records the outcome and evidence for each.
Spikes that remain genuinely open are marked and tracked on the roadmap.

| Spike | Question | Outcome |
|---|---|---|
| SPIKE-001 Upstream reuse audit | Reuse code from GitactionBoard / Git PR Dashboard? | **Resolved — conceptual reuse only.** No code was copied; both projects informed feature behavior (workflow reliability presentation, PR normalization). Recorded in `credits.yaml` with upstream commits; no license text bundling required beyond notices already present. Temporary forks were therefore unnecessary. |
| SPIKE-002 Workers free CPU feasibility | Do webhook verify + parse + D1 writes fit 10 ms CPU? | **Resolved in production.** v0.1.0–v0.2.0 run live on the free plan: Web Crypto signature checks, compact event translation, prepared statements, bounded batch collectors. No CPU-limit errors observed in operation. Guardrails (subrequest caps, checkpoints) implemented as designed. |
| SPIKE-003 GitHub App permission matrix | Exact read-only permission set? | **Resolved.** The working permission set is documented in `docs/setup/github-app.md`; capability detection distinguishes not_authorized / unsupported per feature instead of assuming uniform access. |
| SPIKE-004 REST vs GraphQL | Which collection style fits the budgets? | **Resolved — REST-first.** Bounded REST collectors with rate-limit header capture proved sufficient within free-tier budgets; GraphQL remains an optimization option if subrequest pressure appears at larger estate sizes. |
| SPIKE-005 Branch health semantics | What does "main is current" mean? | **Resolved.** FR-005 semantics implemented in the domain package (`current` / `work_pending` / `untracked_work` / `diverged` / `unknown`) with unit tests (`packages/domain/test/branch-status.test.ts`) and configurable exclusions. |
| SPIKE-006 Budgets and usage coverage | What budget data is readable in practice? | **Partially resolved.** Budget enumeration implemented with capability gating; enhanced-billing usage ingestion remains to be validated against an organization with the enhanced billing platform. Open item on the roadmap. |
| SPIKE-007 GitLab discovery/webhooks | GitLab.com vs self-managed, tokens, hooks? | **Resolved.** Group/subgroup discovery, MRs, pipelines, comparisons, and an optional webhook receiver shipped in v0.2.0; reconciliation is authoritative, webhooks are an optimization. PAT supported; OAuth remains preferred for multi-user futures. |
| SPIKE-008 Dashboard authentication | App OAuth vs Cloudflare Access? | **Resolved — GitHub App user OAuth** with username allowlist and signed HttpOnly session cookies; Cloudflare Access stays optional in front. Implemented in the worker auth module. |
| SPIKE-009 D1 schema and indexes | Schema, indexes, retention cost? | **Resolved.** Migrations 0001/0002 with indexed dashboard/selector queries and retention compaction implemented; row-scan behavior acceptable at current estate scale. Re-benchmark at 1,000+ repositories. |
| SPIKE-010 Webhook idempotency | Duplicates, replays, missed events? | **Resolved.** Delivery-ID (GitHub) and fingerprint (GitLab) idempotency implemented with translator unit tests (`packages/provider-github/test/webhooks.test.ts`); reconciliation repairs missed events; manual redelivery documented in the reconnect runbook. |
| SPIKE-011 UI framework | shadcn/MUI/headless? | **Resolved — lightweight custom.** Hand-rolled CSS + React Router + TanStack Query keeps the bundle ~112 kB gzipped and dependency surface minimal; a component library can be revisited if the admin surface grows. |
| SPIKE-012 Diagram workflow | Mermaid vs Lucid authority? | **Resolved.** Mermaid in Git is canonical (design pack); Lucid is for workshop/stakeholder copies that link back to the source and state authority. |
| SPIKE-013 License/provenance automation | SPDX/REUSE tooling? | **Resolved — lightweight.** `credits.yaml` is the machine-readable source generating the in-product credits page; CI validates it. Full REUSE tooling deferred until third-party code is actually vendored. |
| SPIKE-014 Self-host portability | Do core packages run without Cloudflare types? | **Directionally resolved, unproven.** Domain/contracts/provider packages avoid Cloudflare imports by construction (`persistence-d1` and the worker are the only adapters). A Node/SQLite proof remains future work (Phase 6). |
| SPIKE-015 Notifications and PWA | Connectors, push, offline? | **Partially resolved.** Generic outbound escalation webhook shipped (v0.2.0). Teams/Slack/Discord connectors, acknowledgements/quiet hours, and the PWA shell remain roadmap items (Phase 5 tail). |
